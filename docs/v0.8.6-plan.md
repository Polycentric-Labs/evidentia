## What shipped

### Phase 1 — CIMD scope enforcement at MCP-protocol level + per-call audit trail

- **NEW** `evidentia_mcp.scope` module (~250 LOC) ships
  `enforce_cimd_scope(server, default_client_id)` that
  monkey-binds a wrapper to `FastMCP.call_tool` (mcp Python
  SDK 1.27 has NO public middleware hook; subclass-or-monkey-
  bind is the canonical pattern; `FastMCP.call_tool` at line
  343 is the chokepoint).
- Pass-through path: when `server.evidentia_cimd is None` the
  gate delegates with no audit emit (preserves v0.8.5 default
  no-gating behavior; absence of audit events IS itself an
  audit signal).
- Authorization path: registered `client_id` + scope contains
  `tool_name` → emit `AI_MCP_TOOL_AUTHORIZED` + delegate to
  original `call_tool`.
- Denial paths: ambiguous-caller (no client_id resolvable) +
  unregistered client_id + out-of-scope tool → emit
  `AI_MCP_TOOL_DENIED` + raise `McpError` code -32602
  (Invalid Params per JSON-RPC 2.0).
- Per-call UUID4 `run_id` for audit correlation (distinct
  from DFAH run_id namespace).
- Per-transport client_id resolution: stdio = `--default-
  client-id` IS the client_id (documented as INFORMATIONAL
  audit-trail granularity, NOT a security boundary); HTTP/SSE
  = `Context.client_id` from request meta with `--default-
  client-id` fallback; both None → ambiguous-caller policy →
  DENY.
- 2 new EventActions: `AI_MCP_TOOL_AUTHORIZED` +
  `AI_MCP_TOOL_DENIED`. Documented in `docs/log-schema.md`
  `evidentia.mcp.*` section per the AU-3 contract.
- New `--default-client-id <slug>` CLI flag wired through
  stdio + SSE + HTTP transports. Validation: warn when set
  without `--cimd-registry`; warn when `--cimd-registry` set
  without `--default-client-id` on stdio (every call denies
  per ambiguous-caller policy).
- Examples: `examples/mcp/cimd-registry-readonly.json` +
  `cimd-registry-power.json` operator-friendly samples.
- 8 new unit tests in `tests/unit/test_mcp/test_scope.py`.
- **Threat model**: CIMD is NOT authentication. Documented
  prominently in both `cimd.py` (v0.8.5 P4) + `scope.py`
  (v0.8.6 P1) docstrings + `docs/threat-model.md` v0.8.6
  delta. Operators MUST wire transport-level authentication.

### Phase 2 — Cohen's Kappa rater agreement script + label-quality probe

- Closes the v0.8.5 P2 multi-rater methodology reservation.
- New `scripts/compute_inter_rater_kappa.py` ships Cohen's
  Kappa formula κ = (po - pe) / (1 - pe) + Landis-Koch 1977
  verbal interpretation + CI-gateable exit codes.
- Two operating modes: two-rater file mode (`--rater1` +
  `--rater2` JSONL files) + rule-based-rater mode (`--rule
  jaccard --rule-threshold N`; deterministic; no LLM tokens
  / human time).
- Empirical results against `corpus.jsonl` (51 framework-
  agnostic entries):
  - threshold 0.30: κ = 0.0585 (slight)
  - threshold 0.50: κ = 0.0956 (slight)
  - threshold 0.70: κ = 0.3284 (fair)
  - threshold 0.85: κ = 0.4848 (moderate)
- Best κ (0.4848 at threshold 0.85) below the ≥ 0.80
  acceptance target. Per §29 R3 mitigation, the corpus ships
  as **"single-rater + κ probe inconclusive"** with documented
  rationale that the substantial moderate-to-poor agreement
  empirically demonstrates exactly what motivated the v0.8.3
  P1.1 sentence-transformers semantic faithfulness path.
- New `tests/data/dfah-calibration/inter-rater-agreement.md`
  documents methodology + results + reproduction recipe +
  v0.8.7 / v0.9.0 reservations for real LLM-assisted +
  human-second-rater passes.
- ASCII output (Greek κ spelled out as "kappa") for
  Windows-cp1252 portability.
- 25 new unit tests covering κ formula + Landis-Koch labels +
  rule-based rater.

### Phase 3 — Per-claim bootstrap-resampled confidence + framework-aware threshold defaults

- **NEW** `FaithfulnessResult.confidence: float | None = None`
  field — bootstrap-resampled stddev (default-off cost-aware
  ~100ms/claim; opt-in via `compute_confidence=True` kwarg).
- **NEW** `FaithfulnessResult.framework: str | None = None`
  field — persisted on result for audit-trail re-derivation.
- **NEW** `DEFAULT_THRESHOLDS_BY_FRAMEWORK_JACCARD` constant:
  NIST 0.60 / FFIEC 0.35 / ISO27001 0.30 (per the v0.8.5 P2
  empirical sweep).
- **NEW** `resolve_threshold(framework, method)` helper falls
  back to `DEFAULT_FAITHFULNESS_THRESHOLD` (0.30) for unknown
  frameworks / non-jaccard methods.
- **NEW** `_bootstrap_confidence(claim_tokens,
  clause_token_sets, n_resamples, seed)` helper resamples the
  claim's token set N times (default 100) with replacement;
  returns `1.0 - normalized_stddev` clamped to [0, 1].
- `faithfulness_score()` extended with 4 backward-compatible
  kwargs: `framework=`, `compute_confidence=`, `n_resamples=`,
  `confidence_seed=`.
- 10 new unit tests across 3 test classes (TestConfidence +
  TestFrameworkField + TestResolveThreshold).
- **CLI flag `--faithfulness-threshold-mode {framework-aware,
  fixed}` deferred to v0.8.7** — library + helper shipped in
  v0.8.6 P3; CLI surface lands in v0.8.7 P2.
