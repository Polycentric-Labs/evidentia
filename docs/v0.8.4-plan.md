## Context — v0.8.3.1 hot-fix recap

v0.8.3 attempted G4 via Path 1 (SOURCE_DATE_EPOCH-driven
`uv build` → byte-identical wheels across hosts → matching
SHA256 hashes between local pre-tag pip-compile + PyPI
uploads). v0.8.3 release.yml first-fire revealed `uv build` is
NOT byte-identical between Windows local + Linux CI runner
even with same SOURCE_DATE_EPOCH (file-ordering / timestamp-
precision drift). PyPI publish succeeded but container build's
`pip install --require-hashes` failed: local-computed hashes ≠
Linux-CI-built wheel hashes. **Hot-fix v0.8.3.1** reverted
Dockerfile to exact-version pinning
(`pip install evidentia[gui]==X.Y.Z`) — same v0.8.2 surface,
no regression; container ship recovered same-day. v0.8.4
revisits G4 via Path 2.

## What shipped

### Phase 1 — G4 Path 2 ACTIVATED

- **`release.yml` post-PyPI regeneration step** added to the
  publish-container job BETWEEN the existing Wait-for-PyPI step
  + the docker build step. New step writes
  `docker/requirements.in` containing
  `evidentia[gui]==<tag-version>`, then runs:
  ```bash
  pip-compile --generate-hashes \
              --no-emit-find-links \
              --output-file=docker/requirements.txt \
              docker/requirements.in
  ```
  → ephemeral `docker/requirements.txt` overwrite → docker build
  picks it up. Built-in 3-attempt retry loop with 30s sleeps
  absorbs PyPI propagation lag through the CDN.
- **Dockerfile install line re-flipped** to:
  ```dockerfile
  COPY docker/requirements.txt /tmp/requirements.txt
  RUN pip install --no-cache-dir --user --require-hashes \
      -r /tmp/requirements.txt
  ```
- **Hashes match because pip-compile downloads from PyPI's
  bytes** in the Linux CI runner — same source as the container
  build's pip install. Cross-platform reproducibility no longer
  required. Defense-in-depth: hash verification fires at
  pip-compile time + at install time (two distinct points in
  the supply chain).
- **Committed `docker/requirements.txt` is preview state** for
  operators reading the repo; release-time regeneration
  overwrites it ephemerally before the docker build picks it
  up. Operators inspecting a source checkout see approximate
  hashes from the most recent regeneration; the as-shipped
  hashes are baked into the published container image.
- **Closes recurring Scorecard PinnedDependencies false-positive
  cycle** (alerts #100 → #116 across v0.7.12 → v0.8.3.1)
  structurally + permanently. Score moves from 9/10 → 10/10
  at v0.8.4 ship-time scan.

### Phase 2 — DFAHarness `check_faithfulness=True` wiring

- **`EvalSample` schema** gains optional
  `source_clauses: list[str] | None = None` field (backward-
  compatible — existing samples without source_clauses skip
  faithfulness check even when the harness is asked to run it).
- **`PromptFaithfulnessResult` Pydantic model** with
  `prompt_id: str` + `claims: list[FaithfulnessResult]` +
  computed properties `overall_faithful: bool` (= all claims
  passed) + `passed_count: int` + `failed_count: int`.
- **`EvalResult` schema** gains
  `faithfulness_results: list[PromptFaithfulnessResult]`
  field (default empty list; populated only when harness ran
  with `check_faithfulness=True`).
- **`DFAHarness.run()` extension** — 5 new kwargs:
  - `check_faithfulness: bool = False` (default off)
  - `faithfulness_threshold: float = DEFAULT_FAITHFULNESS_THRESHOLD`
  - `faithfulness_method: Literal["jaccard", "semantic"] = "jaccard"`
  - `claim_extraction_fn: Callable[[str], list[str]] | None = None`
  - `faithfulness_score_fn: Callable[..., FaithfulnessResult] | None = None`

  When `check_faithfulness=True`: for each sample whose
  `source_clauses` is set, the harness extracts atomic claims
  from the post-determinism modal output (matches v0.8.0 P0.1
  review fix F7 canonical replay logic), scores each claim
  against the sample's source_clauses via the chosen method,
  fires `EventAction.AI_EVAL_FAITHFULNESS_CHECKED` per-prompt
  + `EventAction.AI_EVAL_FAITHFULNESS_VIOLATION` per below-
  threshold claim, appends `PromptFaithfulnessResult` to
  `EvalResult.faithfulness_results`.
- **EventAction activations** —
  `AI_EVAL_FAITHFULNESS_CHECKED` (reserved-but-inactive in
  v0.8.0; ACTIVATED in v0.8.4) +
  `AI_EVAL_FAITHFULNESS_VIOLATION` (reserved-but-inactive in
  v0.8.0; ACTIVATED in v0.8.4). Operators get the same audit-
  trail granularity for faithfulness violations that the
  v0.8.0 baseline gave for determinism violations.
- **Mock-callable injection points** — `claim_extraction_fn`
  + `faithfulness_score_fn` keep harness tests cost-zero (no
  LLM or sentence-transformers token burn in CI) while
  exercising real production code paths. Default callable
  resolution falls back to v0.8.3-shipped `extract_claims` +
  v0.8.2/v0.8.3-shipped `faithfulness_score` /
  `faithfulness_score_semantic` when callers don't inject
  mocks.
- **14 new unit tests** across 5 test classes covering: default-
  False / sample-skipping / claim-result aggregation / audit-
  event firing / method-selection / model JSON-roundtrip.
