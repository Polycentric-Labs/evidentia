## Theme

**"UI alpha.2 completion + research-resync + benchmark design +
quickstart polish."**

v0.4.0 shipped the alpha.1 read-only web UI; the interactive
onboarding wizard, Gap Analyze form, Gap Diff picker, and
Risk Generate streaming page were planned for alpha.2 but
never landed (other priorities took precedence through v0.7.x).
v0.7.6 closes that loop — the GUI gets to feature-complete for
the v0.7 minor line — alongside the deferred quarterly research
re-sync (DOC5 from v0.7.3) if the Q3 cadence has arrived, plus
the performance benchmarking design + measurement work, the
90-second quickstart tutorial, and the deep `/security-review`
threat-model walk.

This is a polish + completion release. No new collectors, no new
output integrations, no new catalog tiers. Everything here is
finishing already-claimed surface. Realistic ship target: 4-5
weeks after v0.7.5.

## P0.8 — GitHub Enterprise + Code Security + Secret Protection (NEW 2026-05-01)

Allen requested setting up a free GitHub Enterprise account to access GitHub's enhanced security tooling. Current state:

- **`allenfbyrd/evidentia`** is a public personal-account repo. Several GitHub Advanced Security features are already free for public OSS:
  - ✅ CodeQL code scanning (running)
  - ✅ Dependabot alerts + version updates (running)
  - ✅ Secret scanning (passive — flags committed secrets in history)
  - ✅ OpenSSF Scorecard workflow (already integrated)
- **NOT yet enabled / requires setup**:
  - ⚠️ **Push Protection** — blocks pushes containing detected secrets BEFORE they land in history. Free for public repos as of 2024 but must be explicitly enabled in repo Security settings.
  - ⚠️ **Code Security organization-level dashboard** — requires the repo to live under a GitHub organization, not a personal account.
  - ⚠️ **Custom CodeQL queries / custom security configurations** — requires either GHAS subscription OR a free GitHub Enterprise Cloud trial / Free org tier with Advanced Security enabled.
  - ⚠️ **Branch ruleset enforcement against admins** (currently the v0.7.5 push showed "Bypassed rule violations for refs/heads/main") — requires the repo under an org with rulesets configured.

### Decision points

The phrase "free GitHub Enterprise account" admits two interpretations — surface both to Allen for decision:

| Option | Effort | Effect |
|---|---|---|
| **A. Enable existing-but-unused free features on the personal repo** | ~30 min | Push Protection (Settings → Code security and analysis → Push protection: enabled). Custom CodeQL config (already part of repo via `.github/codeql/codeql-config.yml`). NO migration; repo stays at `github.com/allenfbyrd/evidentia`. |
| **B. Migrate the repo to a new GitHub organization** (e.g., `evidentia-grc/evidentia`) + enable GitHub Advanced Security free trial | ~2-3 hours | Organization-level Code Security dashboard, finer-grained rulesets, eventual Enterprise tier upgrade path. **Cost**: every external link to `allenfbyrd/evidentia` redirects via GitHub's permanent transfer URL — README badges, the resume link Allen has on his personal site, the composite action `uses: allenfbyrd/evidentia/...` references, the PyPI Trusted Publisher claim (would need re-registration per the `evidentia_v0_7_0_review.md` trap memory), every cosign verification regex (`certificate-identity-regexp` baked into `release.yml:251` + sigstore-quickstart.md docs), and external citations of the repo. **Risky and high-effort.** |
| **C. Hybrid** — keep `allenfbyrd/evidentia` as the canonical home; create a `evidentia-grc` org as a thin "stewarding" wrapper with `evidentia` as a fork mirror, and enable Enterprise features on the org. | ~4-5 hours | Best of both: existing URLs work; org-level features available; future migration path is open. **Complexity**: keeping fork + canonical in sync; double the GitHub maintenance surface. |

**Recommended (P2 priority)**: **Option A** for v0.7.6 — enable Push Protection + verify all free features are on. Defer Options B/C to a future milestone if + when the project graduates from "personal portfolio" to "stewarded OSS project" — that's a separate strategic decision well outside v0.7.6 scope.
