## 2. v0.10.3 candidates (ranked by leverage)

### Candidate A — Upstream the marketplace plugin (lowest-effort, high-signal)

- **Scope**: Submit the v0.10.2-staged plugin PR to
  `GRCEngClub/claude-grc-engineering`. Engage-first via issue,
  fork + branch, PR.
- **Effort**: 2-4 hours (engage + PR + maintainer iteration).
- **Surface change**: zero in our repo (plugin already staged); the
  upstream merge is what makes the plugin discoverable to GRC
  Engineering Club marketplace users.
- **Approval gate**: external-repo PR requires explicit per-action
  approval.
- **Best if**: Allen wants the community-discoverability signal
  before continuing further feature work.

### Candidate B — Persona modes (`--persona <name>` CLI switch)

- **Scope**: Add a global `--persona` flag to the CLI; per-persona
  default config (e.g., `--persona engineer` defaults to
  `--format sarif` for `gap analyze`, surfaces engineering-friendly
  output paths). 5 personas: auditor, engineer, TPRM, federal,
  model-risk.
- **OSS-vs-paid scope**: per the standing policy
  (`docs/v0.10.2-marketplace.md` §6), auditor + engineer personas
  are OSS; TPRM / federal / model-risk personas reserved for the
  Pro / Federal commercial tier. **Surfaces an OSS-vs-paid
  decision** at phase entry — recommend defer to v0.11.0 + the
  commercial-tier launch design.
- **Effort**: 1-2 days (CLI surface + per-persona config + tests
  + docs).
- **Best if**: Allen wants UX polish + a clear OSS-vs-paid
  inflection point.

### Candidate C — YAML-driven catalog / control-tier definitions

- **Scope**: Lower the barrier to adding new framework catalogs.
  Today, adding a framework requires a Python PR to
  `evidentia_core.catalogs.data` and a `frameworks.yaml` entry.
  Goal: a fully YAML-driven catalog where contributors drop a
  YAML file into `catalogs/yaml/<framework-id>.yaml` and the
  loader auto-picks it up (with schema validation, tier
  classification, redistribution-license metadata).
- **OSS-vs-paid scope**: clearly OSS — lowering the contributor
  barrier benefits the entire community. No decision needed.
- **Effort**: 2-3 days (YAML schema + loader + 2-3 framework
  migrations as proof + docs).
- **Best if**: Allen wants to invest in community contributor
  ergonomics ahead of the operator deep-dive.

### Candidate D — Map onto OpenSSF Gemara reference model

- **Scope**: Per the integration-survey, map Evidentia's surface
  onto the OpenSSF Gemara reference-model components (security
  signals, evidence collection, attestation, etc.). Doc-only
  update to `positioning-and-value.md` + a new
  `docs/gemara-mapping.md`.
- **OSS-vs-paid scope**: clearly OSS (positioning material).
- **Effort**: 4-8 hours (research + doc).
- **Best if**: Allen wants strategic-positioning polish before
  scaling community engagement.

### Candidate E — Operator deep-dive entry

- **Scope**: NOT a feature release. Allen's planned hands-on
  product-validation pass: work through every CLI verb, REST
  endpoint, MCP tool, UI page; build out the project wiki; produce
  demo recordings.
- **Sequencing**: Allen specified this happens AFTER the v0.10.x
  line is complete. With the original v0.10.x themes (persona
  modes, YAML catalogs, AI-gov packs, Gemara) unshipped, the
  deep-dive is technically premature — but the integration-line
  core (OCSF + SARIF + ingestion + MCP-as-backend + marketplace
  staging) is now solid enough that an early deep-dive could
  surface real backlog priorities for the rest.

## 3. Recommended sequencing

Honest recommendation: **Candidate A first** (upstream the plugin
— quick win, validates the integration-survey's complementary-not-
competitive thesis with real GRC Engineering Club community
feedback), THEN **Candidate E** (operator deep-dive — the v0.10.2
ship has compressed enough integration-line surface into a usable
product that hands-on validation will surface the right v0.10.4+
priorities better than continuing speculative feature work).

Candidates B + C + D can be re-prioritized after the deep-dive
based on what gaps the deep-dive surfaces.

## 4. Out of scope (deferred further)

- v1.0 acceptance gates (multi-reviewer walkthroughs, OpenSSF
  Gold, cryptographic CIMD signatures wired at FastMCP dispatch)
  per the v1.0 master plan.
- Pro / Federal commercial tier launch — post-v1.0.
- TPRM mutation MCP tools (`poam_create`, vendor risk-tier update,
  vendor onboarding) — deferred from v0.10.2.

## 5. Release mechanics

Whatever v0.10.3's scope ends up being, the bump shape is
**patch** unless Candidate B's persona modes ship in full (in
which case minor → v0.11.0 is defensible). Allen's standing
preference for full /pre-release-review applies regardless.
