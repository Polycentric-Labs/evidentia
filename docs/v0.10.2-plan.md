## 1. Context

v0.10.0 + v0.10.1 shipped the OCSF interop foundation:
schema-aligned `SecurityFinding`, bidirectional mapping, SARIF
output, third-party ingestion collector. The next move on the
v0.10.x line is the **MCP-as-backend + GRC Engineering Club
marketplace plugin** (per [`integration-survey.md`](integration-survey.md)
§7 and yesterday's research recommendation).

Two complementary deliverables:

1. **Deepen Evidentia's MCP tool surface** so AI clients (Claude
   Desktop, Claude Code, custom MCP clients) can drive the full
   product end-to-end — not just the 8 v0.9.7 tools (list_frameworks
   / get_control / gap_analyze / gap_diff / conmon_*).
2. **Publish a thin Evidentia plugin into the GRC Engineering Club's
   `claude-grc-engineering` marketplace** — a curated, named entry
   point so the largest GRC-engineering-via-Claude community can
   discover and use Evidentia natively in their Claude Code
   workflows.

Plus one optional hardening item from the v0.10.1 review:

3. **F-V101-L1 (LOW) SSRF hardening on `collect ocsf` URL mode** —
   optional `--block-private-ips` flag.

## 2. Phase 1 — MCP tool surface expansion

Audit the existing 8 frozen MCP tools (per `docs/api-stability.md`
§MCP tool contract) and identify which Evidentia capabilities are
NOT yet exposed:

- `evidentia collect aws|github|okta|sql|databricks|snowflake|vanta|
  drata|bitsight|securityscorecard|ocsf` — none of these collectors
  has an MCP tool wrapper.
- `evidentia gap analyze --format sarif` — SARIF output isn't
  exposed.
- `evidentia tprm vendor list|risk-tier|...` — TPRM surface absent.
- `evidentia poam ...` — POA&M surface absent.
- `evidentia model-risk ...` — AI model-risk surface absent.

Prioritize **the highest-leverage 3-5 additions** rather than
trying to expose everything. Candidates ranked by integration
value (per the research pass):

- **`gap_analyze_sarif`** — gap-as-CI-gate from inside the AI
  client. (Tier 1.)
- **`collect_ocsf`** (file mode only — URL mode out of scope for
  MCP for the same trust-boundary reason as F-V101-L1). Lets an AI
  client ingest Prowler / Security Hub OCSF output and surface it
  through `gap_analyze`. (Tier 1.)
- **`tprm_vendor_list`** + **`tprm_vendor_risk_tier`** — third-
  party risk surface that VANTA / Drata already expose. (Tier 2.)
- **`poam_list`** + **`poam_create`** — POA&M lifecycle for the
  federal-tier persona. (Tier 2.)

Each new tool follows the existing pattern in
`packages/evidentia-mcp/src/evidentia_mcp/tools/` — `@mcp.tool()`
decorator + typed input/output models + CIMD scope guards.

## 3. Phase 2 — GRC Engineering Club marketplace plugin

Per yesterday's research, `GRCEngClub/claude-grc-engineering` is a
Claude Code plugin marketplace with persona plugins, framework
plugins, connectors, OSCAL tooling, etc. The natural Evidentia
entry: publish a **thin plugin** that wires the Evidentia MCP
server into a Claude Code session with a curated set of skills /
prompts oriented around the GRC Engineering workflows.

Sub-deliverables:

- **`evidentia-mcp` Claude Code plugin manifest** —
  pointer at `evidentia mcp serve` with a default tool allowlist;
  documents the install via `claude-grc-engineering`'s marketplace
  conventions.
- **2-3 curated skills** that wrap common Evidentia flows
  (e.g. "run gap analysis on my repo's evidence inventory and emit
  SARIF for the next PR", "ingest a Prowler output file and
  cross-walk to NIST 800-53").
- **PR to upstream `claude-grc-engineering`** adding the Evidentia
  plugin entry to their marketplace registry. **Requires Allen's
  explicit approval** per the publishing-authority protocol (it's
  an external-repo PR).

Acceptance gate: an operator who installs the
`claude-grc-engineering` marketplace and enables the Evidentia
plugin can run gap analysis + OCSF ingestion + SARIF emit from
within Claude Code without separate CLI invocations.

## 4. Phase 3 — F-V101-L1 SSRF hardening (optional)

Add a `--block-private-ips` flag (default `True` would be safer
default; `False` for operators who explicitly need to ingest from
internal endpoints) to `evidentia collect ocsf` URL mode. Rejects
URLs that resolve to RFC1918 + 169.254/16 + 127/8 ranges BEFORE
opening the connection (DNS resolution via `socket.gethostbyname`,
range check, then `urllib.request.urlopen`).

Acceptance gate: an attempt to ingest from `https://169.254.169.254/`
(AWS metadata) with the default flag set raises `OCSFIngestError`
with a clear message; explicit opt-out via `--no-block-private-ips`
allows the operator to override for trusted internal use.

## 5. Sequencing & dependencies

```
Phase 1 (MCP tool surface) ──┬─> Phase 2 (GRC Eng Club plugin)
                             └─> [independent]
Phase 3 (SSRF hardening) ────────> [independent of Phase 1 + 2]
```

Phase 2 depends on Phase 1 (at minimum the new MCP tools that the
plugin will surface). Phases 1 + 3 are independent of each other.

## 6. Risks & mitigations

| Risk | Mitigation |
|---|---|
| MCP tool surface grows faster than CIMD scope can keep up | Every new tool ships with a documented CIMD scope grant; default scope = OSS-tier read-only operations. |
| GRC Engineering Club PR rejected or has conflicting marketplace conventions | Engage their maintainers via a draft PR / issue first; align on naming + scope before shipping. **OSS-vs-paid scope decision surfaces here**: do persona-tied skills (e.g. "TPRM vendor onboarding") stay OSS or move to Pro? Surface to Allen at Phase 2 entry. |
| SSRF hardening (Phase 3) breaks legitimate internal-endpoint ingestion use cases | Make the flag default-False initially (warn but allow), flip to default-True in v0.11.0 after one cycle. OR ship default-True with explicit `--no-block-private-ips` opt-out. Decision in Phase 3 design. |

## 7. Open-source vs paid scope — surfaces in Phase 2

Phase 2's "curated skills" raise the first real v0.10.x OSS-vs-paid
decision: the persona-tied skills (TPRM, federal, model-risk) might
naturally fit the Pro/Federal commercial tier rather than the OSS
marketplace plugin. Allen to decide at Phase 2 entry.

## 8. Out of scope (deferred further)

- Persona modes proper (`--persona auditor|engineer|tprm`) — later
  v0.10.x.
- YAML-driven catalog / control-tier definitions — later v0.10.x.
- AI-governance regulatory packs (EU AI Act Annex IV, ISO/IEC
  42001) — later v0.10.x.
- Map onto OpenSSF Gemara reference model — positioning material,
  not feature.

## 9. Release mechanics

v0.10.2 is a patch bump from v0.10.1 if Phase 3 is the only
substantive change; minor bump if Phase 1's new MCP tool surface +
Phase 2's plugin land together (those constitute new feature
surface). Decide at Step 5 of the v0.10.2 ship cycle.
