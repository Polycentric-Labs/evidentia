## P0.6 — Model Risk Management overlay (`evidentia model-risk`)

New top-level capability module per SR 11-7 + OCC Bulletin 2011-12
(historical structure) + SR 26-02 + OCC Bulletin 2026-13a (April
2026 active guidance). The 2026 guidance EXPLICITLY EXCLUDES
generative AI + agentic AI from scope — banks running LLM-driven
controls have no governing regulatory framework. v0.7.10's model-
risk module addresses both the active SR 26-02 / OCC 2026-13a stack
AND positions Evidentia's `GenerationContext` provenance chain as
the SR-replacement framework for the generative-AI gap.

### P0.6.1 — Model inventory schema

Pydantic model in `evidentia_core.models.model_risk.ModelInventory`:

| Field | Type | Notes |
|---|---|---|
| `id` | `str` (UUID v4) | Stable identifier; auto-generated |
| `name` | `str` | Model name |
| `purpose` | `str` | Business purpose (per SR 11-7 §III.A "Conceptual Soundness") |
| `methodology` | `Methodology` enum | `statistical` / `ml` / `rules_based` / `llm` / `expert_judgment` / `hybrid` — maps to validation requirements |
| `vendor_or_internal` | `Provenance` enum | `internal` / `vendor` — internal models follow §III; vendor models follow §V |
| `vendor_id` | `str \| None` | Cross-link to TPRM `Vendor.id` when `vendor_or_internal=vendor` |
| `tier` | `Tier` enum | `tier_1` / `tier_2` / `tier_3` per SR 11-7 criticality framework (Tier 1 = high-impact + complex) |
| `owner` | `str` | Internal model owner (email or LDAP identifier) |
| `inputs` | `list[ModelInput]` | Data sources, transformations, lineage |
| `outputs` | `list[ModelOutput]` | Decisions affected, downstream consumers |
| `last_validation_date` | `date \| None` | |
| `validation_findings` | `list[ValidationFinding]` | Open + closed findings with severity |
| `next_validation_due` | `date \| None` | Auto-computed from tier + last_validation_date (Tier 1 → annual; Tier 2 → biennial; Tier 3 → triennial) |
| `retirement_plan` | `str \| None` | Per SR 11-7 §III.C ongoing monitoring |
| `evidence_refs` | `list[EvidenceRef]` | Reuses the v0.7.9 TPRM EvidenceRef model |
| `created_at` / `updated_at` / `evidentia_version` | auto | Standard Evidentia provenance |

Persistence mirrors `vendor_store.py`: one JSON file per model
under platformdirs-backed user-dir; `EVIDENTIA_MODEL_STORE_DIR`
env-var override; atomic `os.replace` save semantics.

### P0.6.2 — Model documentation generator

Per SR 11-7 §III.A "Conceptual Soundness" + §III.B "Ongoing
Monitoring" + §III.C "Outcomes Analysis", generates model
documentation templates in Markdown + XLSX + OSCAL formats:

CLI: `evidentia model-risk doc generate --model-id <id> --format {md,xlsx,oscal} --output <path>`

Output sections per SR 11-7:

- §III.A.1 Model purpose + intended use
- §III.A.2 Theory + design + data
- §III.A.3 Implementation + numerical methods
- §III.B.1 Outcomes analysis (back-testing + benchmarking)
- §III.B.2 Process verification (sensitivity + bias)
- §III.C Ongoing monitoring (performance + change management)
- Appendix A: Data lineage (auto-populated from `inputs` + the
  v0.7.7/v0.7.8 SQL/cloud-DW collectors if linked)
- Appendix B: Validation history (auto-populated from
  `validation_findings`)
- Appendix C: Linked evidence (auto-populated from
  `evidence_refs`)

XLSX output gated behind the existing `[xlsx]` extra (added in
v0.7.9 P0.2 second slice). OSCAL output emits a custom OSCAL
component-definition with model-as-component framing.

### P0.6.3 — Validation report templates

Per SR 11-7 §III.D + Annex matrix, generates validation report
scaffolding for an independent model validator to fill in:

CLI: `evidentia model-risk validation-report --model-id <id> --findings findings.json --output report.md`

Inputs:

- `--model-id`: the ModelInventory record under review
- `--findings`: optional JSON file of pre-existing
  ValidationFinding entries (e.g., previously surfaced gaps)
- `--output`: target Markdown / XLSX file

Output: scaffolded report with sections for the validator to
fill in (the specific test cases, sensitivity probes, back-test
tables) but with the inventory metadata + mapped controls
pre-filled.

### P0.6.4 — AI-feature linkage

The Evidentia AI features (`evidentia risk generate`, `evidentia
explain`) ARE model invocations under SR 11-7. v0.7.10 wires every
risk-statement run to a model-inventory entry:

- Add `model_inventory_ref: str | None` field on `RiskStatement`
- The existing `GenerationContext` (model, temperature,
  prompt_hash, run_id, evidentia_version) ALREADY captures SR
  11-7 §III.A.3 "implementation" evidence — v0.7.10 documents the
  mapping in `docs/model-risk.md`.
- Sigstore-sign the `(risk_statement, model_inventory_ref)`
  link so the SR-11-7 audit trail is cryptographically
  verifiable.
- New CLI: `evidentia model-risk link --risk-id <r> --model-id <m>`
- Doc cross-reference: when a `RiskStatement` carries a
  `model_inventory_ref`, the OSCAL AR emit (extending v0.7.9
  P0.5) includes the linked ModelInventory in `back-matter.resources`.

### P0.6.5 — REST surface

Mirrors the v0.7.9 P0.1 TPRM REST CRUD pattern:

- `POST /api/model-risk/models` — create
- `GET /api/model-risk/models?skip=0&limit=100` — paginated list
- `GET /api/model-risk/models/{model_id}` — get one
- `PUT /api/model-risk/models/{model_id}` — replace
- `DELETE /api/model-risk/models/{model_id}` — delete
- `GET /api/model-risk/models/{model_id}/next-validation-due` —

- `POST /api/model-risk/models/{model_id}/doc` — generate doc
- `POST /api/model-risk/models/{model_id}/validation-report` —
  generate report

All endpoints follow the v0.7.8 F-V08-DAST-3 invariant (400 not
422 for body errors).

## P1.5 — Governance primitives (G1 + G2 only; G3-G5 to v0.7.11)

Per the comprehensive plan §19.1, the 5 governance primitives
(originally scoped for the 8-10 week v0.7.9 plan) split:
v0.7.10 takes G1 + G2; v0.7.11 takes G3 + G4 + G5.

### G1 — Three Lines of Defense / Three Lines Model (IIA) capability

Per the Institute of Internal Auditors' Three Lines Model (2020
update of the original Three Lines of Defense framework):

- **First Line**: management + operational functions that own and
  manage risk (CISO / CIO / CDO equivalents)
- **Second Line**: independent risk + compliance oversight
  functions providing effective challenge to the first line
- **Third Line**: internal audit providing independent assurance
  to governing bodies + senior management

**New schema**: `evidentia_core.models.governance.LineOfDefense`
enum + `Owner` Pydantic model with `line_of_defense` field. Every
control assignment + risk statement + finding can be tagged with
which line owns/oversees it.

**Capability**: `evidentia governance lines-report` — produce a
matrix showing controls + risks + findings stratified by line of
defense, useful for audit-committee + board-level reporting.

### G2 — Effective Challenge primitives (2LOD evaluation workflow)

Effective challenge is the second line's regulatory obligation to
independently evaluate first-line risk assessments and push back
when standards aren't met. v0.7.10 ships primitives for documenting
+ tracking effective-challenge events:

- New schema: `evidentia_core.models.governance.EffectiveChallenge`
  with fields for the first-line assessment under review, the
  second-line evaluator, the challenge basis (citation to standard
  / framework / internal policy), the outcome (accepted / modified
  / escalated), and Sigstore-signed evidence of the evaluation
- CLI: `evidentia governance challenge log --first-line-assessment
  <ref> --basis "NIST 800-53 RA-3" --outcome modified --notes "..."`
- Automatic linkage to gap-analysis findings + risk statements so
  each finding carries the chain of effective challenges that
  shaped its remediation plan

## Strategic positioning (carryforward from v0.7.9)

The v0.7.10 capability surface continues the v0.7.9 financial-
services positioning:

- **Vendor SaaS GRC** (Vanta, Drata, AuditBoard / Optro,
  MetricStream, ServiceNow IRM): mostly framework-coverage +
  workflow tools, do NOT ship SR 26-02 / OCC 2026-13a / FFIEC
  catalogs as bundled out-of-the-box content
- **Open-source GRC ecosystem** (CISO Assistant, Eramba, Comp AI,
  compliance-trestle): primarily focused on NIST 800-53 / ISO
  27001 / SOC 2; the financial-services regulatory stack is
  underserved
- **Evidentia v0.7.10 positioning**: the only OSS GRC tool that
  ships TPRM + Model Risk + FFIEC + OCC + SR 11-7/SR 26-02 as
  first-class bundled capabilities, with cryptographic evidence
  chain-of-custody + air-gap support + SR-replacement-grade
  audit evidence for LLM-driven controls (the regulatory gap
  the 2026 guidance explicitly leaves open).
