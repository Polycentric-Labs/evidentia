- 4 new flags on `evidentia eval risk-determinism`:
  `--check-faithfulness`, `--faithfulness-threshold N`,
  `--faithfulness-method {jaccard,semantic}`,
  `--source-clauses-file <yaml>`.

  without `--source-clauses-file`; unknown faithfulness
  method; malformed YAML at top level; non-list[str] entry
  values.

  prompt_id matches the gap's `<framework>:<control_id>`
  (or `<framework>:<control_id>#N` for disambiguation).

  threshold; total claims scored + prompt count;

Closes the v0.8.4 P1.2 CLI-surface deferral. Library +

### Phase 2 — DFAH calibration corpus expansion

- Corpus growth 51 → 123 entries via 3 new JSONL files:
  - `corpus_nist.jsonl` (24 entries) — NIST 800-53 control
    text shapes (account review, audit retention,
    cryptographic keys, vulnerability scanning, session
    timeout, backup recovery)
  - `corpus_ffiec.jsonl` (24 entries) — FFIEC IT Examination
    Handbook + OCC bulletin shapes (board oversight,
    independent audit, third-party risk, business continuity,
    customer authentication, encryption in transit, incident
    reporting)
  - `corpus_iso27001.jsonl` (24 entries) — ISO 27001:2022
    Annex A shapes (security objectives, risk treatment,
    personnel screening, removable media, supplier
    relationships, cryptographic review, performance
    evaluation)
- Each new entry carries a `framework` field (`"nist-800-53"`
  / `"ffiec-it-handbook"` / `"iso-27001"`) for downstream
  filtering.
- `scripts/tune_faithfulness_threshold.py --corpus-pattern
  <glob>` flag for per-framework sweep. Sweeps each matching
  JSONL file separately and reports per-file recommended
  threshold + Youden's J.
- Empirical per-framework sweep (jaccard scorer):
  - `corpus_ffiec.jsonl`: threshold=0.35, J=0.417
  - `corpus_iso27001.jsonl`: threshold=0.30, J=0.417
  - `corpus_nist.jsonl`: threshold=0.60, J=0.417
- Multi-rater methodology section added to corpus README:
  single-rater (Allen) baseline + LLM-assisted generation +
  manual spot-check on ~20% of new entries. Cohen's Kappa
  target ≥ 0.80 (deferred to v0.8.6).

### Phase 3 — real-LLM integration tests

- `tests/integration/test_eval/test_real_llm_extraction.py`
  with 4 tests (3 LLM-burning + 1 ungated empty-input edge
  case).
- LLM-burning tests gated by `EVIDENTIA_LLM_INTEGRATION=1`
  env var; CI never runs them automatically.
- Tests assert STRUCTURAL properties (claim count,
  per-claim token count, score distribution trend) rather
  than exact-match strings — different LLM models produce
  different splits.
- Score-distribution trend test: faithful entries score
  higher than unfaithful on average. The canonical "scorer
  works" sanity check.
- Cost expectation: ~5-10 LLM calls × ~$0.001/call ≈ $0.005-
  $0.05 per full integration run with gpt-4o-mini.

### Phase 4 — MCP CIMD richness

After 5 deferral cycles (v0.8.0 → v0.8.4), Allen's "implement
now" lock-in ends the deferral pattern.

- New module `evidentia_mcp.cimd`:
  - `CIMDDocument` Pydantic model — one client's metadata
    per OAuth Dynamic Client Registration spec (RFC 7591) +
    MCP authentication conventions. Fields: `client_id`
    (1-200 chars), `client_name` (1-500 chars), `scope`
    (space-separated tool allowlist), `redirect_uris`,
    `policy_uri`, `tos_uri`. `has_scope(tool_name)`
    implements deny-by-default allowlist semantics (empty
    scope = deny-all).
  - `CIMDRegistry` Pydantic model — version-tagged registry
    of CIMDDocuments loaded from JSON via
    `CIMDRegistry.from_file()`. Validates top-level JSON
    object + version match (current: 1) + per-CIMDDocument
    Pydantic validation. Wraps malformed-input errors into
    `ValueError` with operator-friendly messages.
- Wired into MCP server: `build_server()` + `run_stdio()` +
  `run_sse()` + `run_http()` accept optional `cimd_registry=`
  kwarg. Server attaches the registry as
  `server.evidentia_cimd` for tool implementations that opt
  into manual scope checks.
- Wired into CLI: `evidentia mcp serve --cimd-registry <path>`
  flag. Loader errors surface as exit 2 with explicit
  messages.
- 19 new unit tests in `tests/unit/test_mcp/test_cimd.py`
  across 5 test classes (CIMDDocument model + CIMDRegistry
  from_file + lookup + build_server attachment + CLI flag).

v0.8.5 ships the registry-loading + attachment
infrastructure. **Per-tool scope enforcement at MCP-protocol
level (rejecting tool calls when client_id lacks scope) is
deferred to v0.8.6** — the registry IS visible to tool
implementations via `server.evidentia_cimd` for callers
that opt into manual scope checks.

**Threat model**: CIMD is NOT authentication — it's a
metadata + scope layer that runs ON TOP of whatever
authentication the transport provides. Operators deploying
CIMD MUST also wire transport auth (reverse-proxy mTLS or
bearer tokens) so clients cannot impersonate each other's
CIMD entries. Documented prominently in `cimd.py` docstring.
